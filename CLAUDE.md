# CLAUDE.md — Komplid Marketing Agent Directives

Проект: **komplid-marketing** — маркетинговый сайт komplid.ru на Next.js 15.
Связь с приложением: **app.komplid.ru** (репозиторий `stroydocs/` — отдельный, не трогаем).
Цель: SEO + AEO + GEO-трафик для B2B и B2C-аудитории (Профи-пакеты для сметчиков, ПТО, прорабов).
Юрформа: **ИП на УСН «Доходы» 6% без НДС** (выручка до 20 млн ₽/год освобождена).
Стек: Next.js 15 App Router · TypeScript · Tailwind · shadcn/ui · MDX · Docker · Timeweb Cloud VDS.
Дизайн-эталон: `design/landing-reference.html` (из Claude Design, адаптирован под план).
Полный план реализации — в `MODULE_MARKETING_PLAN.md`.

## 2. ДИЗАЙН-ЭТАЛОН — ЕДИНСТВЕННЫЙ ИСТОЧНИК ПРАВДЫ

Файл `design/landing-reference.html` — **единственный источник правды** для дизайна.
Он содержит полный одностраничный лендинг с палитрой, токенами OKLch, типографикой,
блоками Hero/Pricing/FAQ/Footer и всеми актуальными текстами.

**Правила работы с дизайном:**

1. **Перед созданием любого нового компонента** — открой `landing-reference.html`
   и посмотри есть ли там похожая секция. Если есть — копируй HTML оттуда, не придумывай с нуля.

2. **OKLch-токены в `:root` блоке лендинга** переносятся в `src/styles/globals.css`
   один-в-один. Это гарантирует визуальное единство с приложением app.komplid.ru.

3. **Палитры:** основная — Steel (`--accent: oklch(0.72 0.16 62)`), резервные Cobalt и Lime.
   Все три определены в лендинге через `html[data-palette="..."]`.

4. **Light/Dark темы** через `html[data-theme="..."]`. Переключатель в Header.

5. **Шрифты:** Inter (UI) + JetBrains Mono (eyebrow, код, метки). Подключаются через
   `next/font/google` в `src/app/layout.tsx`.

6. **Не использовать Tailwind-цвета напрямую** (`text-gray-500`, `bg-blue-100` и т.п.).
   Вместо этого — `text-[var(--ink-soft)]`, `bg-[var(--bg-inset)]`.

Если что-то не совпадает с эталоном → сначала правишь эталон, потом код.
Никогда наоборот.

## 3. ЧТЕНИЕ ФАЙЛОВ ЧАНКАМИ

Файлы >500 строк читать ТОЛЬКО чанками — offset + limit параметры.
Особенно актуально для `landing-reference.html` (1500+ строк), `MODULE_MARKETING_PLAN.md`
(3400+ строк), больших MDX-статей.

```
Файл < 500 строк  → один read, можно доверять
Файл 500–2000 строк → читать чанками по 400–500 строк
Файл > 2000 строк → читать только нужный раздел, явно указывать offset
```

Tool results >50 000 символов обрезаются молча. Если grep по контенту статей
вернул подозрительно мало — перезапусти с более узким scope.

## 4. ПОСЛЕ 10+ СООБЩЕНИЙ — ПЕРЕЧИТЫВАТЬ ФАЙЛЫ

После 10 и более сообщений в сессии ОБЯЗАТЕЛЬНО перечитать файл перед правкой.
Авто-компакция могла молча уничтожить контекст — редактирование по памяти = сломанный код.

Признаки деградации контекста:
- Ты предлагаешь компонент который уже есть в `src/components/`
- Ты «помнишь» статью которой нет в `content/blog/`
- Правка не совпадает с реальным содержимым файла

При обнаружении → перечитай файл, сообщи: «Перечитываю после компакции».

## 5. БЕЗОПАСНОСТЬ ПРАВОК

**Перед каждым edit → перечитай файл.** Edit tool молча падает если old_string не совпадает.
**После каждого edit → перечитай файл** и убедись что изменение применилось.
Не делай более 3 правок одного файла без промежуточного чтения.

При переименовании компонента/типа/страницы искать ВСЁ:
- Прямые imports в `src/`
- Ссылки в MDX-файлах `content/`
- Упоминания в sitemap.ts и robots.ts
- URL в HTML-лендинге (`design/landing-reference.html`)
- Внутренние ссылки в других статьях (`/blog/old-slug`)
- Редиректы в `next.config.mjs`

Grep не гарантирует полноту — assume it missed something.

## 6. ФАЗНОЕ ВЫПОЛНЕНИЕ

Многофайловые задачи — только фазами. Максимум 5 файлов на фазу.
Выполнить фазу → верификация (`npx tsc --noEmit`) → ждать подтверждения → следующая фаза.

Когда пользователь говорит «сделай план» — только план, без кода.
Когда пользователь говорит «да», «делай», «го» — выполняй без повторения плана.

При создании новой посадочной (например `/smetchik`) фазы такие:
1. Страница + metadata
2. Schema.org разметка (SoftwareAppSchema + BreadcrumbSchema + FaqSchema)
3. Блоки контента (Hero / Features / Pricing / FAQ)
4. Внутренние перелинковки (на /blog, другие посадочные)

## 7. ПРАВИЛА SEO / AEO / GEO ДЛЯ ВСЕХ СТАТЕЙ И ПОСАДОЧНЫХ

Это **главное** отличие маркетинг-сайта от приложения. Каждая страница должна
одновременно ранжироваться в Яндекс/Google (SEO), цитироваться в AI-ответах
Алисы/Нейро/Google Overview (AEO) и упоминаться в ChatGPT/Perplexity/Claude (GEO).

### 7.1. Прямой ответ в первых 2-3 предложениях

ПЛОХО:
> «В этой статье мы рассмотрим, как правильно заполнить АОСР...»

ХОРОШО:
> «АОСР — акт освидетельствования скрытых работ, обязательный документ ИД.
> Актуальная форма установлена приказом Минстроя РФ №344/пр от 2015 года
> с поправками 2023. Бесплатный шаблон ниже.»

AI-системы забирают прямой ответ как цитату. Вода в начале = потеря AEO-сигналов.

### 7.2. H2-подзаголовки в форме вопросов

ПЛОХО: `## Общая информация об АОСР`
ХОРОШО: `## Что такое АОСР и для чего он нужен?`

AEO-боты сканируют H2 как вопросы и сопоставляют с запросами пользователей.

### 7.3. FAQ-секция в конце статьи — ВСЕГДА

Минимум 5-7 вопросов с форматированием `**Вопрос?**` → ответ.
Извлекается автоматически через `extractFaqFromContent()` и рендерится в
`FaqSchema` (JSON-LD разметка).

Без FaqSchema статья не попадёт в AI Overview. Это некомпромиссное правило.

### 7.4. Frontmatter обязательные поля

```yaml
---
title: "..."              # SEO title, 50-60 символов
description: "..."         # meta description, 150-160 символов
slug: "..."
publishedAt: "2026-05-01"
author: "Komplid"
tags: [...]                # 3-5 тегов
primaryQuestion: "..."     # главный вопрос статьи для AEO
keyTakeaway: "..."         # короткий ответ, показывается в блоке "Главное"
readingTime: 5
---
```

Без `primaryQuestion` и `keyTakeaway` статью не принимать в main.

### 7.5. Внутренние перелинковки — минимум 3

Каждая статья должна ссылаться на минимум 3 другие страницы сайта:
- На посадочные `/smetchik`, `/pto`, `/prorab` (где уместно)
- На другие статьи блога (`/blog/other-slug`)
- На шаблоны `/shablony/aosr` или калькуляторы `/kalkulyator/...`

Цель: укрепление семантического графа сайта для Google/Яндекса.

### 7.6. Авторитетные внешние ссылки

Минимум 1 ссылка на нормативный документ или авторитетный источник:
- Минстрой РФ (minstroyrf.gov.ru)
- Ростехнадзор (gosnadzor.ru)
- ГОСТ (protect.gost.ru)
- Реестр НОСТРОЙ (reestr.nostroy.ru)

AI-системы учитывают outbound-качество при оценке доверия к странице.

### 7.7. Callout и ComparisonTable — AEO-магниты

В каждой длинной статье должно быть:
- Минимум 2 `<Callout>` с ключевыми фактами (type="info" | "warning" | "tip" | "success")
- Минимум 1 `<ComparisonTable>` или `<StepList>` если применимо

AI-системы обожают структурированный контент — выдёргивают его дословно.

### 7.8. Длина — не меньше 1500 слов для серьёзной статьи

Короткие статьи (<800 слов) не ранжируются по коммерческим запросам.
Исключение: страницы шаблонов (landing-подобные, короткие).

## 8. ПРАВИЛА РАБОТЫ С КОНТЕНТОМ (MDX)

### 8.1. Структура контент-папок

```
content/
├── blog/              → статьи (slug.mdx)
├── shablony/          → страницы бесплатных шаблонов
├── sravneniya/        → сравнения с конкурентами (cus.mdx, exon.mdx)
└── authors/           → авторы (пока только Komplid)
```

### 8.2. Добавление новой статьи — процесс

1. Создать файл `content/blog/{slug}.mdx` с обязательным frontmatter
2. Скопировать шаблон статьи из `content/blog/_template.mdx` (если есть)
3. Следовать правилам раздела 7 (SEO/AEO/GEO)
4. Проверить: FAQ в конце (минимум 5 вопросов), перелинковки (минимум 3),
   Callout (минимум 2), внешняя ссылка (минимум 1)
5. Если статья упоминает конкурента (ЦУС, Exon) — **обязательно** сослаться
   на страницу сравнения `/sravnenie/cus` или аналогичную

### 8.3. Компоненты MDX (импортируются автоматически из `mdx-components.tsx`)

- `<Callout type="info|tip|warning|success" title="...">` — выделенный блок
- `<StepList>` + `<Step number={1} title="...">` — пошаговая инструкция
- `<ComparisonTable columns={["A", "B"]} rows={[...]}>` — таблица сравнения
- `<DownloadCard>` — карточка скачивания шаблона
- `<Quote>` — цитата клиента или эксперта

Не использовать кастомные HTML-теги — только эти компоненты.

## 9. ЗАПРЕЩЁННЫЕ РЕШЕНИЯ (не предлагать!)

- ❌ **Tilda** для лендингов или блога. Всё на Next.js. Обоснование в MODULE_MARKETING_PLAN.md раздел 0.
- ❌ **GitHub Actions** для CI/CD. Деплой через локальный `./deploy.sh`.
- ❌ **Вложение в stroydocs/**. `komplid-marketing` — полностью отдельный репо.
- ❌ **НДС 22% в чеках или офертах**. Мы ИП на УСН без НДС. В счетах «НДС не облагается».
- ❌ **Реквизиты ООО** в футере, оферте, чеках. Только ИП (ФИО, ОГРНИП, ИНН 12-значный).
- ❌ **localStorage/sessionStorage** в MDX-статьях. Работает сборка через SSG.
- ❌ **Client-side запросы к api.komplid.ru напрямую**. Только через `src/app/api/*/route.ts`
  с `INTERNAL_API_TOKEN` в header.
- ❌ **Импорт компонентов из stroydocs/**. Нет shared-library, они всегда отдельные.
- ❌ **Английский текст в контенте**. Только русский. Исключение — URL-слаги на латинице.
- ❌ **Emoji в навигации и footer**. Эмодзи допустимы в статьях блога для эмфазы.

## 10. ПРАВИЛА КОДА ПРОЕКТА

- TypeScript strict mode — никаких `any`
- Комментарии к бизнес-логике — **на русском**
- Название бренда в коде: **Komplid** (не komplid, не KOMPLID)
- Домен в коде: `komplid.ru` (маркетинг), `app.komplid.ru` (приложение)
- Все внешние запросы — через `next/server` route handlers, никаких fetch из клиентских компонентов
- Компоненты: максимум 150 строк, сверх — разделять на под-компоненты
- Разделение UI и логики (обязательно):
  - `*.tsx` — только JSX + минимальный стейт
  - `use*.ts` — сложная логика (хотя на маркетинг-сайте такого почти нет)
- Путь к @: `@/*` → `./src/*`, `@/content/*` → `./content/*` (настроено в tsconfig)
- Файлы статических изображений в `public/images/` с осмысленными именами
- SVG-иконки через `lucide-react`, не через raw SVG (кроме блоков из дизайн-эталона)

## 12. СВЯЗЬ С app.komplid.ru — ТОЛЬКО ЧЕРЕЗ API

Маркетинг-сайт НЕ имеет доступа к БД приложения. Единственный канал — `INTERNAL_API_URL`
(публичное API в stroydocs). Используется для:

- Отправка лидов (форма скачивания шаблона, подписка на рассылку, заявка на демо)
- Валидация реферального кода на странице `/ref/[code]`
- Получение публичных метрик («152 объекта на платформе») — опционально

**Паттерн:**
```typescript
// src/app/api/lead/route.ts (только серверный код!)
const res = await fetch(`${process.env.INTERNAL_API_URL}/leads`, {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'Authorization': `Bearer ${process.env.INTERNAL_API_TOKEN}`,
  },
  body: JSON.stringify(parsed.data),
});
```

**Запрещено** вызывать `INTERNAL_API_URL` из клиентских компонентов — токен утечёт.

Все CTA на покупку/регистрацию ведут на `https://app.komplid.ru/signup` с UTM-метками:
```
?utm_source=komplid_ru&utm_medium=organic&utm_campaign={article_slug}&plan={plan}
```

предотвратить этот класс ошибок. Не просто исправляй — делай post-mortem.

## 15. ВАЖНЫЕ ТЕХНИЧЕСКИЕ РЕШЕНИЯ

1. **SSG по умолчанию** — все страницы должны билдиться статически. `generateStaticParams`
   для динамических роутов (`/blog/[slug]`, `/shablony/[slug]`, `/sravnenie/[slug]`).
   Исключение: `/api/*` роуты — force-dynamic.

2. **Метаданные через Next.js Metadata API** — не использовать раздельные `next/head`.
   Все `metadata` и `generateMetadata` — в `page.tsx` или `layout.tsx`.

3. **JSON-LD через `<Script type="application/ld+json">`** — не через `<meta>`.
   Компоненты в `src/components/seo/*Schema.tsx` уже готовые.

4. **Изображения через next/image** с формата AVIF + WebP. Никаких `<img>` кроме OG-превью.

5. **Фото статей — в `public/images/blog/{slug}.jpg`**, размер ~1200×630 (OG-формат).

6. **Шаблоны скачивания — в `public/shablony-files/*.docx`**.
   API `/api/template-download` отдаёт их с правильными заголовками.

7. **Реферальные коды** — приходят из URL `/ref/[code]`, сохраняются в cookie на 30 дней,
   прокидываются в UTM при клике на CTA.

## 16. ТИПОВЫЕ ПРОМПТЫ — ЧАСТЫЕ ЗАДАЧИ

Готовые промпты для повторяющихся операций. Пользователь может прислать их
копипастом — выполняй по шаблону ниже.

### 16.1. «Создай новую статью о {тема}»

Шаги:
1. Прочитай CLAUDE.md (пункты 7 и 8 про SEO-правила и структуру MDX)
2. Создай `content/blog/{slug-kebab-case}.mdx` со всеми обязательными frontmatter-полями
3. Структура: прямой ответ (3 абзаца) → 5-7 H2-подзаголовков в форме вопросов →
   минимум 2 Callout → минимум 1 ComparisonTable или StepList → FAQ-секция (6+ вопросов)
4. Минимум 3 внутренние перелинковки + 1 авторитетная внешняя ссылка
5. Добавь картинку-заглушку в `public/images/blog/{slug}.jpg` (можно пустой файл пока)
6. Проверь что `/blog/{slug}` открывается в dev (`docker compose up`)
7. Запусти `npx tsc --noEmit`

### 16.2. «Создай посадочную для {роль}»

Шаги:
1. Прочитай `design/landing-reference.html`, посмотри на секцию ProfiPackagesTeaser
2. Создай `src/app/{role}/page.tsx` (например `/smetchik/page.tsx`)
3. Обязательные элементы:
   - `metadata` с title, description, canonical, keywords
   - `<SoftwareAppSchema>` + `<BreadcrumbSchema>` (в начале page)
   - `<Hero>` с прямым ответом в subtitle (что это, для кого, от какой цены, триал)
   - `<Features>` — 6 фичей именно под эту роль
   - `<ComparisonTable>` — сравнение с существующими решениями (Excel/Гранд/ЦУС)
   - `<Pricing>` — B2C тарифы с тумблером Месяц/Год
   - `<Faq>` — 7-8 вопросов специфичных для роли
   - `<Cta>` финальный с UTM
4. UTM-метки на всех CTA: `?utm_source=landing&utm_campaign={role}&role={role}`
5. Добавь новую посадочную в `sitemap.ts` (приоритет 0.9)
6. Запусти `npx tsc --noEmit`

### 16.3. «Создай страницу шаблона {название}»

Шаги:
1. Создай `content/shablony/{slug}.mdx` с frontmatter (title, slug, filename, format, size, regulation, category)
2. Положи файл в `public/shablony-files/{filename}.docx`
3. Описание шаблона в MDX: что включено, инструкция, нормативная ссылка
4. Related templates: 2-3 связанных шаблона
5. CTA внизу: «Хотите автоматизировать? → /pto» (или /smetchik в зависимости от типа)

### 16.4. «Создай сравнение с {конкурент}»

Шаги:
1. Создай `content/sravneniya/{slug}.mdx`
2. Структура: короткий ответ в Callout → обзор Komplid → обзор конкурента →
   большая ComparisonTable на 20-25 строк → «кому подойдёт Komplid» →
   «кому подойдёт конкурент» → FAQ (6+ вопросов) → финальный CTA
3. Честность — обязательно. Признавать сильные стороны конкурента (опыт, референсы).
4. **Не указывать конкретные цифры конкурентов** без ссылки на источник.

### 16.5. «Задеплой текущие изменения на прод»

Это делает пользователь, а не Claude Code. Но проверь перед деплоем:
1. `npx tsc --noEmit` — ноль ошибок
2. `docker compose up --build` — сайт локально работает
3. Проверить в браузере что новые страницы открываются
4. Проверить Rich Results Test (mentally) для любых новых JSON-LD разметок

После этого пользователь сам запускает `./deploy.sh`.
